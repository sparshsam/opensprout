# v0.8.1 — MCP Server for AI Agents

> **For Hermes:** Use subagent-driven-development to implement this plan task-by-task.

**Goal:** Ship an MCP server as part of OpenSprout so AI agents (Claude Code, Hermes, Cursor, etc.) can read a user's plant data and perform actions on their behalf.

**Architecture:** A standalone TypeScript MCP server package (`apps/mcp/`) using `@modelcontextprotocol/sdk`. Agents connect via stdio transport and authenticate with a Supabase access token passed in the `authorization` header. The server delegates reads and writes to the Supabase client with RLS-enforced user-scoping — no service role key, no multi-tenant bypass. Ships as part of the monorepo; deployable as a standalone Node process or on Vercel serverless/Edge Functions.

**Tech Stack:** TypeScript, `@modelcontextprotocol/sdk`, `@supabase/supabase-js`, Node.js 20+

**Auth model:** Users generate a Personal Access Token from OpenSprout settings. Agents present it as a Supabase-compatible JWT in the MCP session. The server validates the token and creates a Supabase client scoped to that user. Every query and mutation is guarded by RLS — the same policies that protect the web app protect agent access.

## Task 2: Supabase client factory with token validation

**Objective:** Create a function that takes an auth token string, validates it against Supabase, and returns an RLS-scoped Supabase client scoped to that user's data. No service role key — agents work within the same permission model as web users.

**Files:**
- Create: `apps/mcp/src/supabase.ts`
- Modify: `apps/mcp/package.json` (add dotenv for local dev)

**`src/supabase.ts`:**
```typescript
import { createClient } from "@supabase/supabase-js";
import type { Database } from "../../apps/web/src/lib/data/types.js";

const SUPABASE_URL =
  process.env.NEXT_PUBLIC_SUPABASE_URL ?? process.env.SUPABASE_URL ?? "";
const SUPABASE_ANON_KEY =
  process.env.NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY ??
  process.env.SUPABASE_ANON_KEY ??
  "";

export function createSupabaseClient(token: string) {
  if (!SUPABASE_URL || !SUPABASE_ANON_KEY) {
    throw new Error(
      "Missing Supabase credentials. Set NEXT_PUBLIC_SUPABASE_URL and NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY."
    );
  }

  const client = createClient<Database>(SUPABASE_URL, SUPABASE_ANON_KEY, {
    global: {
      headers: {
        Authorization: `Bearer ${token}`,
      },
    },
  });

  return client;
}

export async function getAuthUser(token: string) {
  const client = createSupabaseClient(token);
  const {
    data: { user },
    error,
  } = await client.auth.getUser(token);
  if (error || !user) {
    throw new Error(`Authentication failed: ${error?.message ?? "Invalid token"}`);
  }
  return user;
}
```

The server reads `NEXT_PUBLIC_SUPABASE_URL` and `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY` from the environment — same env vars the web app uses. Agents and users configure these when setting up the MCP server.

**Key auth flow:**
1. Agent starts MCP session, passes `authorization: Bearer <user-token>` via environment or tool call header
2. Server calls `supabase.auth.getUser(token)` to validate
3. All subsequent Supabase queries use that token-bound client, scoped by RLS

**Pattern for each tool (plants.ts example):**

```typescript
import type { SupabaseClient } from "@supabase/supabase-js";
import type { Database } from "../../apps/web/src/lib/data/types.js";

export function registerPlantTools(server: McpServer, getClient: () => SupabaseClient<Database>) {
  server.tool(
    "list_plants",
    "List all plants in the user's collection with their species, health status, and location.",
    {},
    async () => {
      const { data, error } = await getClient()
        .from("plants")
        .select("*")
        .is("deleted_at", null)
        .order("name");

      if (error) throw error;
      return { content: [{ type: "text", text: JSON.stringify(data, null, 2) }] };
    }
  );

  server.tool(
    "get_plant",
    "Get detailed information about a specific plant by its ID.",
    {
      plantId: z.string().describe("The ID of the plant to retrieve"),
    },
    async ({ plantId }) => {
      const { data, error } = await getClient()
        .from("plants")
        .select("*")
        .eq("id", plantId)
        .is("deleted_at", null)
        .single();

      if (error) throw error;
      if (!data) return { content: [{ type: "text", text: "Plant not found" }] };
      return { content: [{ type: "text", text: JSON.stringify(data, null, 2) }] };
    }
  );
}
```

The key design choice: each tool receives `getClient` (a lazy factory that creates a Supabase client bound to the authenticated user's token). This keeps auth centralized.

```typescript
server.tool(
  "log_care_activity",
  "Log a care activity (e.g., watering, fertilizing) for a plant. Creates a care_log record and updates the task if a taskInstanceId is provided.",
  {
    plantId: z.string().describe("The plant ID"),
    careType: z.enum(["water", "fertilize", "mist", "rotate", "prune", "repot", "inspect", "custom"]),
    notes: z.string().optional().describe("Optional notes about the care activity"),
    taskInstanceId: z.string().optional().describe("Optional task instance ID to mark complete"),
  },
  async (args) => {
    const client = getClient();
    const { data, error } = await client.from("care_logs").insert({
      plant_id: args.plantId,
      user_id: user.id,
      care_type: args.careType,
      notes: args.notes ?? null,
    }).select().single();

    if (error) throw error;

    // If a task was specified, mark it complete
    if (args.taskInstanceId) {
      await client.from("task_instances").update({
        status: "done",
        completed_at: new Date().toISOString(),
      }).eq("id", args.taskInstanceId);
    }

    return { content: [{ type: "text", text: JSON.stringify(data, null, 2) }] };
  }
);
```

The `identify_plant` tool reuses the existing PlantNet proxy pattern: accepts a base64 image, sends to the identify-plant Edge Function, returns results. This avoids duplicating the PlantNet API key logic.

## Task 6: Access token management in the web app

**Objective:** Add a Settings UI page where users can create and revoke MCP Personal Access Tokens. Tokens are stored as hashed values in a new `mcp_tokens` Supabase table.

**Files:**
- Create: `supabase/migrations/XXXXXX_add_mcp_tokens.sql`
- Create: `apps/web/src/lib/data/mcp-tokens.ts`
- Create: `apps/web/src/app/(authenticated)/settings/mcp/page.tsx`
- Modify: `apps/web/src/app/(authenticated)/settings/page.tsx` (add link)

**Supabase migration:**
```sql
create table public.mcp_tokens (
  id uuid default gen_random_uuid() primary key,
  user_id uuid not null references auth.users(id) on delete cascade,
  name text not null,
  token_hash text not null,
  token_prefix text not null,  -- first 8 chars for display (osp_abc123...)
  last_used_at timestamptz,
  created_at timestamptz default now(),
  revoked_at timestamptz
);

create index idx_mcp_tokens_user on public.mcp_tokens(user_id);
alter table public.mcp_tokens enable row level security;

-- Users can only see their own tokens
create policy "Users manage their own tokens"
  on public.mcp_tokens
  for all
  using (auth.uid() = user_id)
  with check (auth.uid() = user_id);
```

**Token generation flow:**
1. User clicks "Create Token" in Settings
2. App generates a cryptographically random token with prefix `osp_`
3. Full token shown to user ONCE in a dialog ("copy this, you won't see it again")
4. SHA-256 hash stored in DB
5. User sets up MCP server with the full token

**Settings page:**
- List existing tokens (name, prefix, last used, created date)
- "Revoke" button per token (sets `revoked_at`)
- "Create New Token" button → dialog with generated token

## Risks, tradeoffs, and open questions

### Risks
1. **MCP SDK version drift** — the `@modelcontextprotocol/sdk` is pre-1.0. Pin the version and lock it down. If the SDK changes its API between now and the next Hermes release, our tools may break.
2. **Token UX** — the "show token once" pattern is essential for security but can confuse users. The Settings UI must be crystal clear ("Copy this now. You won't see it again.")
3. **Supabase token JWT expiry** — Supabase access tokens expire (default 1 hour). The MCP server runs as a long-lived process, so the token could expire mid-session. Options: use a refresh token flow (complex) or generate longer-lived custom tokens from Supabaase (better — the `mcp_tokens` table stores hashes, and the server validates against a custom auth endpoint).

### Tradeoffs
1. **Stdio vs SSE** — Starting with stdio only (simplest, works with all major AI agents). SSE can be added later if remote access is needed.
2. **Token in env var** — Less ergonomic than passing it per-request, but simpler to implement and more secure (never in command-line args). All major MCP clients support env vars in config.
3. **TypeScript vs Python** — TypeScript keeps the stack homogeneous. The Python MCP SDK is equally mature, but maintaining two language runtimes for one feature adds overhead.

### Open questions for Sparsh
1. **Token expiry** — Should tokens be permanent (until revoked) or time-limited (30/90 days)? If permanent, a compromised token means an attacker has indefinite access.
2. **Publishing** — Should `@opensprout/mcp` be published to npm for `npx -y` install, or should users point their agents at the repo/dist directly? The plan assumes local-path usage with a future npm publish.
3. **Rate limiting** — Should MCP tool calls count toward usage analytics/API rate limits? PlantNet calls in particular cost money per request.
4. **v0.8.1 tag** — Will this be a release branch from `v0.8.0` tag or from `main` (which now has the route.ts fix)? Recommend branching from `main` at `d8e327d`.
